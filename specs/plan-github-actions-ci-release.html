Purpose

Add CI/CD to Nocturne via GitHub Actions: fast PR checks (debug build, JVM unit tests, lint) on every pull request, and an on-demand / tag-driven workflow that produces signed release APKs attached to GitHub Releases. Since the project has no release signing config yet, this plan also establishes one — a keystore kept out of git, wired through keystore.properties locally and GitHub secrets in CI.

Problem

The repo (swbain/minimal-tv-launcher) has no .github/ directory: nothing verifies that a PR compiles, passes testDebugUnitTest, or is lint-clean before merge. There is also no way to produce an installable release build — app/build.gradle.kts declares no signingConfigs, so :app:assembleRelease currently emits an unsigned APK that Android refuses to install. The gitignore does not protect keystore files, so naive signing setup risks committing secrets.

Solution

Three phases, in dependency order:

1) Release signing foundation. Generate a release keystore once (kept outside the repo, backed up by the user). Teach app/build.gradle.kts to read signing credentials from a gitignored keystore.properties at the repo root; when the file is absent (fresh clones, PR CI) the release build type simply stays unsigned, so nothing breaks for anyone without keys. Store the keystore (base64) and its passwords as GitHub Actions secrets.

2) PR checks workflow (ci.yml): one Ubuntu job per run — :app:assembleDebug :app:testDebugUnitTest :app:lintDebug — the exact quality gates the README documents. Gradle caching via gradle/actions/setup-gradle, concurrency-cancel superseded runs, upload test/lint reports on failure. Instrumented tests are deliberately excluded (they need a TV emulator; per CLAUDE.md, UI verification is journey-based and local).

3) Release workflow (release.yml): triggered by pushing a v* tag or manually via workflow_dispatch. Reconstructs the keystore from secrets, runs :app:assembleRelease, verifies the signature with apksigner, uploads the APK as a build artifact, and — on tag runs — creates a GitHub Release with the APK attached as nocturne-<tag>.apk.

2. Generate the release keystore (one-time, user-run)

  -keystore ~/keystores/nocturne-release.jks \

  • [] User backs the keystore file up somewhere durable (it is unrecoverable if lost)

3. Optional signing config in app/build.gradle.kts

Read keystore.properties from the repo root at configuration time; absence is not an error (fresh clones and PR CI build unsigned). Compatible with the enabled configuration cache — file reads at configuration time are tracked inputs.

import java.util.Properties

val keystorePropertiesFile = rootProject.file("keystore.properties")
val releaseSigningAvailable = keystorePropertiesFile.exists()
val keystoreProperties = Properties().apply {
  if (releaseSigningAvailable) keystorePropertiesFile.inputStream().use(::load)
}

android {
  signingConfigs {
    if (releaseSigningAvailable) {
      create("release") {
        storeFile = rootProject.file(keystoreProperties.getProperty("storeFile"))
        storePassword = keystoreProperties.getProperty("storePassword")
        keyAlias = keystoreProperties.getProperty("keyAlias")
        keyPassword = keystoreProperties.getProperty("keyPassword")
      }
    }
  }
  buildTypes {
    release {
      if (releaseSigningAvailable) {
        signingConfig = signingConfigs.getByName("release")
      }
      // existing minify/proguard lines unchanged
    }
  }
}

[x] Phase 2: PR Checks Workflow

One workflow, one job. The three gates share a single Gradle invocation so configuration, daemon, and cache warm-up are paid once. JDK 17 matches the project's jvmToolchain(17) (foojay resolver would fetch it anyway, but matching avoids a download).

1. Author .github/workflows/ci.yml

name: CI

on:
  pull_request:
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  checks:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-java@v5
        with:
          distribution: temurin
          java-version: 17
      - uses: gradle/actions/setup-gradle@v4
        with:
          cache-read-only: ${{ github.ref != 'refs/heads/main' }}
      - name: Build, unit tests, lint
        run: ./gradlew :app:assembleDebug :app:testDebugUnitTest :app:lintDebug
      - name: Upload reports
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: reports
          path: |
            app/build/reports/
            app/build/test-results/
  • [x] Create the workflow file exactly as above (adjust action majors only if a run flags a deprecation)
  • [x] cache-read-only guard: only main pushes write the Gradle cache, PRs read it — prevents cache-thrash from parallel PRs

[wip] Phase 3: Signed Release Workflow

Runs on v* tags (the release path) and on manual dispatch (the dry-run path — same build, artifact only, no GitHub Release). CI reconstructs the keystore from secrets into files that Phase 1's Gradle logic already understands, so the build script needs zero CI-specific code.

1. Author .github/workflows/release.yml

name: Release

on:
  push:
    tags: ['v*']
  workflow_dispatch:

permissions:
  contents: write   # create the GitHub Release

jobs:
  release:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-java@v5
        with:
          distribution: temurin
          java-version: 17
      - uses: gradle/actions/setup-gradle@v4
        with:
          cache-read-only: true

      - name: Materialize signing config
        env:
          KEYSTORE_B64: ${{ secrets.RELEASE_KEYSTORE_B64 }}
          KEYSTORE_PASSWORD: ${{ secrets.RELEASE_KEYSTORE_PASSWORD }}
          KEY_ALIAS: ${{ secrets.RELEASE_KEY_ALIAS }}
          KEY_PASSWORD: ${{ secrets.RELEASE_KEY_PASSWORD }}
        run: |
          echo "$KEYSTORE_B64" | base64 -d > release.jks
          cat > keystore.properties <<EOF
          storeFile=release.jks
          storePassword=$KEYSTORE_PASSWORD
          keyAlias=$KEY_ALIAS
          keyPassword=$KEY_PASSWORD
          EOF

      - name: Build signed release APK
        run: ./gradlew :app:assembleRelease

      - name: Verify signature
        run: |
          "$ANDROID_HOME"/build-tools/36.0.0/apksigner verify --print-certs \
            app/build/outputs/apk/release/app-release.apk

      - name: Name the APK
        run: |
          VERSION="${GITHUB_REF_NAME}"
          [ "${GITHUB_REF_TYPE}" = "tag" ] || VERSION="manual-${GITHUB_SHA::7}"
          cp app/build/outputs/apk/release/app-release.apk "nocturne-${VERSION}.apk"
          echo "APK=nocturne-${VERSION}.apk" >> "$GITHUB_ENV"

      - uses: actions/upload-artifact@v4
        with:
          name: ${{ env.APK }}
          path: ${{ env.APK }}

      - name: GitHub Release
        if: github.ref_type == 'tag'
        uses: softprops/action-gh-release@v2
        with:
          files: ${{ env.APK }}
          generate_release_notes: true

Questionables

APK vs AAB — plan builds an APK. Right call?

Assumption: APK. Nocturne is a sideloaded TV launcher with no Play Store presence; AABs are only useful for Play publishing (and would require Play App Signing). A raw signed APK is directly installable via adb install on the TV. If Play publishing ever happens, add a parallel bundleRelease job then.

Release trigger: tag push + manual dispatch — or something else?

Assumption: both v* tags and workflow_dispatch. Tags are the canonical "cut a release" signal and map 1:1 to GitHub Releases; manual dispatch gives a free dry-run path that exercises the whole signing pipeline without publishing anything. Alternatives rejected: building a signed APK on every main push (wasteful, no versioning story) and GitHub's "release published" event (requires creating the release first, inverting the flow).

Keystore custody — who holds it, what if it leaks or is lost?

Assumption: user-held keystore + GitHub repo secrets is acceptable for this project. The keystore lives in ~/keystores/ and a password manager backup; CI gets a base64 copy via repo secrets. If lost: a new keystore means a new signing identity — every installed device must uninstall/reinstall (no user data of consequence in a launcher, so tolerable but annoying). If leaked: an attacker could sign look-alike updates; rotate by generating a new key and accepting the reinstall. For a personal sideloaded app this risk profile is fine; a hardware token or KMS would be overkill.

versionCode/versionName stay manual (1 / "1.0") — automate from the tag?

Assumption: manual for now. versionCode=1, versionName="1.0" are hardcoded in app/build.gradle.kts. Android updates require a strictly increasing versionCode, so the user must bump it per release — the README release section will say so. Deriving versionName from the git tag (and versionCode from run number or commit count) is a clean follow-up but adds moving parts; deliberately out of scope here.

Release build keeps isMinifyEnabled = false?

Assumption: yes, unchanged. Turning on R8 for a Hilt + SQLDelight + OkHttp app means a proguard-rules shakedown that deserves its own tested change, not a rider on CI setup. APK size is a non-issue for a sideloaded launcher.

No emulator/instrumented tests in CI — is PR coverage enough?

Assumption: yes. CLAUDE.md mandates journey-based UI testing on a live TV emulator, driven interactively — not Espresso in CI. connectedDebugAndroidTest on a hosted runner would need an ARM/ATV emulator boot (~5-10 min, flaky, no TV system images in the standard reactivecircus matrix). JVM unit tests + lint + compile are the right CI envelope for this project today.

Action version pins are majors (@v5, @v4, @v2) — SHA-pin instead?

Assumption: major-version pins are acceptable. SHA-pinning (with Dependabot to advance the pins) is the hardened posture, but for a personal project the maintenance cost outweighs the supply-chain risk. The one secret-touching third-party action is softprops/action-gh-release — widely used; swap for a raw gh release create run-step if that dependency feels wrong. Exact majors current as of authoring; the first live run will surface any deprecation warnings.

Branch protection: should CI be a required check?

Open — user decision, manual step. Once ci.yml has run at least once, GitHub Settings → Branches → protect main → require the checks job. Not scriptable into this plan without admin API calls; noted here so it isn't forgotten. Optional for a solo repo where PRs merge via Graphite anyway.

Design decisions

  • Single Gradle invocation in CI (assembleDebug testDebugUnitTest lintDebug) rather than three parallel jobs: this project is one module; parallel jobs would each pay checkout + JDK + Gradle configuration and compile the same code three times. One job is both faster wall-clock and 3× cheaper in runner minutes.
  • The build script knows nothing about CI. The release workflow fabricates exactly the two files a local signed build uses (release.jks, keystore.properties). One code path, testable locally, no System.getenv forks in Gradle.
  • Signing config is optional, not required. assembleRelease without keys yields an unsigned APK instead of a configuration error — contributors and PR CI never trip over missing secrets.
  • Secrets never echo: the materialize step writes files from env:-mapped secrets via a heredoc whose delimiter is deliberately unquoted so $KEYSTORE_PASSWORD etc. expand — do not "fix" it to <<'EOF', which would write the literal variable names into keystore.properties and break signing. GitHub masks secret values in logs regardless. Caveat: passwords containing $, backslash, or backticks need care (shell expansion in the heredoc plus Java Properties.load backslash-escaping) — simplest is to avoid those characters when generating the keystore passwords.
  • Graphite workflow respected: all pushes/PRs for these changes go through gt submit per repo convention — the plan's validation commands assume the PR already exists via Graphite.
