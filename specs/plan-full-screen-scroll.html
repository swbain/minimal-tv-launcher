PR #2 — Full-screen scroll surface

Purpose

Match the updated TV Launcher.dc.html design in the claude.ai/design project: the vertically scrollable container must cover the entire screen (not just the region below the header), and the time/weather header must scroll with the content instead of staying pinned. The fixed wallpaper + scrim stage stays put behind the scrolling content, exactly as the design's absolutely-positioned background layers do.

Problem

LauncherScreen currently lays out a Column with global padding: a static NocturneHeader on top, then a Box(weight = 1f) holding the LazyVerticalGrid. Consequences that diverge from the design:

  • The scrollable viewport is only the sub-region below the header — rows clip against an invisible line mid-screen instead of sliding under the top edge of the display.
  • The header never moves: scrolling a long grid keeps ~90dp of screen permanently spent on the clock.
  • Grid content is vertically centered (Alignment.CenterVertically) inside the sub-region, while the updated design flows content from the top of the scroll surface.
Before — split layout
Column(padding 50/37dp)
├─ NocturneHeader   ← pinned
└─ Box(weight 1f)
   └─ LazyVerticalGrid  ← scrolls
      (centered vertically)
After — one scroll surface
LazyVerticalGrid(fillMaxSize,
  contentPadding 50/37/50/60dp)
├─ item(span = maxLineSpan)
│  └─ NocturneHeader ← scrolls away
└─ app cards…        ← top-flow

Solution

Collapse the split layout into a single full-screen LazyVerticalGrid and make the header its first full-span item (item(span = { GridItemSpan(maxLineSpan) })). Move the old Column padding into the grid's contentPadding so cards scroll under the screen edges (design padding 74px 100px 120px at 1920×1080 ⇒ 37dp top / 50dp horizontal / 60dp bottom). Drop the vertical centering so content flows from the top. Loading / empty states keep the old header-on-top arrangement — with nothing to scroll, pinned vs. in-flow is indistinguishable. While here, sync NocturneBackground's scrim stops with the updated design source (its KDoc promises an exact mirror). Verify on a headless TV emulator with the instrumented suite plus an android journey D-pad run, then ship via Graphite with a detailed PR report.

Notes

Design source of truth

claude.ai/design project 50077339-6d83-4127-8716-f956abd14bd2, file TV Launcher.dc.html (caption: "full-screen scroll · header scrolls away · arrow keys / WASD to move focus, focus keeps in view"). Key structure: three absolutely-positioned background layers (wallpaper, linear scrim, radial vignette) under a position:absolute; inset:0; overflow-y:auto scroll surface whose padded content holds the header and the 4-column grid. The Android analogue: NocturneBackground (fixed stage, already wraps the screen in MainActivity) + a full-screen LazyVerticalGrid with the header as a full-span item.

Design-px → dp conversion

TokenDesign (1920×1080 px)App (dp @ xhdpi)
Scroll content padding74px 100px 120px37 / 50 / 60 (keep current 50dp gutter)
Header → grid gap70px35dp = 15dp item padding + 20dp arrangement
Grid gap40px20dp (unchanged)
Keep-in-view pad90pxhandled by Lazy grid bring-into-view (see below)

Decisions & tradeoffs

  • Header as a grid item, not a nested scroll column: a full-span item in the existing LazyVerticalGrid gives the scroll-away behavior with zero new nesting, keeps lazy composition for large app lists, and keeps the D-pad focus engine's built-in bring-into-view (the design's keepInView hand-roll is the web stand-in for exactly this).
  • Vertical centering removed: the previous under-full-grid centering came from the old design pass; the updated design flows content from the top under the header. Preview renders will confirm an under-full grid still looks intentional.
  • Loading/empty keep a static header: those states have no scrollable content, so putting the message inside a lazy grid would add complexity for no observable difference.
  • Scrim sync included: the design's scrims changed slightly alongside the scroll rework (bottom stop 0.9, vignette recentered at 40% height). NocturneBackground's KDoc claims an exact mirror of the design source, so shipping the scroll change without the scrim sync would silently break that contract.
  • Focus search up from row 1: with the header unfocusable, D-pad-up from the top row is a no-op (focus stays), which matches TV convention; the design's WASD handler behaves the same way.

Risks

  • Autofocus timing: the first card is now grid index 1 (after the header item). The FocusRequester must stay attached to the first app card; the existing grid_isDpadDrivable test guards the regression.
  • Scroll position on app-list refresh: onResume → ScreenResumed re-emits the list; keyed items (componentName) preserve scroll/focus state, same as today.
  • Headless journey flakiness: journeys on emulators can be slow to settle; prefer explicit waits on visible text ("9:42"-style clock is dynamic — assert on stable anchors like weather/app labels where possible).
