Purpose

Bring the Nocturne home screen up to the finalized option 3a handoff: replace the small circular long-press action row with an anchored menu card (Move · Hide from favorites · Open app settings · Uninstall · Cancel), add a D-pad-driven Move mode that reorders the favorites grid with animated reflow and persists the order to the launcher DB, and fix the cold-start glitch where remote focus briefly flashes onto the settings gear before the app grid appears by fading in all home content together once apps have loaded.

Problem

  • No reorder. The home grid is sorted alphabetically (AppsRepository.loadApps()) with no way to arrange favorites; nothing persists a custom order. Users want to drag a tile around the grid.
  • The long-press menu is the old shape. ui/AppActionMenu.kt renders a three-item circular row (Hide · Uninstall · Cancel). The finalized design is a text-list menu card anchored to the right of the tile with five items — adding Move and Open app settings.
  • Focus flash on cold start. During AppsUiState.Loading, MessageScreen("Loading apps…") renders the header with a focusable GearButton. TV focus lands on the gear (the only focusable target), then jumps to the first card once AppsUiState.Ready arrives and AppGrid fires firstItemFocus.requestFocus() — a visible flash to the settings icon before apps appear.

Solution

Three cohesive slices, shipped as a Graphite stack (each branch builds green and is independently reviewable). The MVI-lite shape is preserved throughout: new intent enters as LauncherActions, one render input stays LauncherState, one-shot effects stay LauncherEvent.

  1. reorder/1-content-fade-in Content fade-in & focus fix. Stop rendering a focusable gear during Loading; reveal the whole Ready surface (gear + clock + weather + grid) with a single alpha fade, and only request grid focus once content is shown. Pure UI, no new state.
  2. reorder/2-order-persistence Order persistence + reorder logic. New SQLDelight appOrder table + OrderRepository, folded into deriveAppsState() so home & settings render in the stored order. Add a pure, unit-tested reorder algorithm and the MoveApp / ReorderStep / CommitMove actions + a reordering field on state. No UI entry point yet — fully JVM-testable.
  3. reorder/3-menu-card-move-mode Menu card + Move mode UI. Rewrite AppActionMenu as the anchored five-item card; wire "Open app settings" to a platform intent and "Move" into Move mode; render the locked-tile treatment, bottom hint pill, and D-pad reorder handling with Modifier.animateItem() reflow.

UI is verified per project convention with Android journeys (no Compose/Espresso tests); logic is covered by JVM unit tests in app/src/test.

  • [] Add ReorderMath.kt with enum class MoveDirection { Left, Right, Up, Down } and: fun moveWithinVisible(visible: List<String>, moving: String, dir: MoveDirection, columns: Int): List<String> — compute source index i; target = i-1/i+1/i-columns/i+columns; block at row edges & list bounds (Left blocked when i % columns == 0, Right when i % columns == columns-1 or i == lastIndex, Up when i < columns, Down when i + columns > lastIndex); on a blocked/no-op move return the list unchanged; otherwise remove at i and insert at the clamped target. Questionables

Stack granularity: 3 branches vs. 1 PR

The repo already ships features as Graphite stacks and the user said "stack that diff if we need." Chosen: a 3-branch stack (fade-in → persistence/logic → menu-card/UI) because each slice builds green and reviews cleanly on its own, and Phase 2 (pure logic + DB) carries almost all the test weight independent of the UI. If Graphite submission fails in this environment, the fallback is one branch (reorder-apps-move-mode) with three commits and a single PR — no code changes needed, just a different submit.

Order model: full package ordering vs. visible-only

Persisting a full ordering of all installed packages (with hidden apps threaded through in place) directly satisfies the handoff rule "hidden apps keep a stable position; moving reorders only the visible sequence." A visible-only list would lose hidden apps' slots when they're re-favorited. Cost: one extra flow in the combine and the threadVisibleIntoFull step — both cheap and unit-tested.

"Open app settings": real intent vs. prototype stub

The handoff says the prototype stubs this but production should fire Settings.ACTION_APPLICATION_DETAILS_SETTINGS. We implement the real intent — it's a few lines in MainActivity, matches the existing uninstall-intent pattern, and ScreenResumed already reconciles state on return. Low risk, correct behavior.

Up/Down = ±COLUMNS, and no-wrap edges

Design specifies Left/Right = ±1 and Up/Down = ±4 (one row), all blocked at grid edges with no wrapping. We hard-code the delta to the grid's COLUMNS (4) rather than measuring actual laid-out rows; this is exact for a fixed 4-column grid. If the grid ever becomes responsive, the delta must come from the real column count — flagged here.

Move mode as a state field vs. an Overlay

Move mode is not a dimming overlay — it runs on the live home surface — so it's modeled as LauncherState.reordering: AppInfo? rather than an Overlay variant. This keeps the existing scrim-owning Overlay semantics intact and lets the grid keep normal rendering while a key handler captures the D-pad.

Journey/emulator availability in an automated pipeline

UI truth for this project is Android journeys on a TV emulator (per CLAUDE.md), which needs a running emulator (started raw+headless under sg kvm per project memory). In a headless build→PR pipeline the emulator may be unavailable; those steps are then marked [f] with the authored journey XML attached, and the gradle gates (assemble/lint/unit) remain the automated must-pass bar.

Reorder algorithm sketch

// visible = current favorites in grid order; full = all installed in stored order
fun onReorderStep(dir: MoveDirection) {
  val moving = state.reordering?.packageName ?: return
  val visible = (state.apps as Ready).apps.map { it.packageName }
  val newVisible = moveWithinVisible(visible, moving, dir, COLUMNS)
  if (newVisible == visible) return                 // blocked edge → no write
  val full = (state.apps as Ready).allApps.map { it.packageName }
  orderRepository.setOrder(threadVisibleIntoFull(full, newVisible))  // persists each step
}
// threadVisibleIntoFull: hidden apps stay pinned; visible slots receive newVisible in sequence

Risks / watch-list

  • Focus during reflow. Re-sorting the backing list while a tile is "held" can bounce D-pad focus; pin focus to the moving package after each step (mirror MenuFocusRestorer's withFrameNanos {} + requestFocus()).
  • Persist-per-step churn. Every ReorderStep writes the full order in a transaction; fine at ~20 apps. If it ever feels heavy, debounce the write on CommitMove (but the design wants each step committed).
  • COLUMNS single-source. The ViewModel and grid must share the column count; moving the constant out of LauncherScreen.kt avoids drift.
  • Schema. Adding a table to an existing on-device DB relies on Schema.create for fresh installs; if the app is already installed with the old schema, SQLDelight needs a migration file. For this launcher (dev installs) a fresh install is expected, but note it before shipping to real devices.
