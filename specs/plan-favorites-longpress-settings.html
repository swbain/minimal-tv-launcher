Purpose

Give Nocturne its curation layer, exactly as specified by design handoff option 2c: the home grid becomes a favorites-only surface; a long-press on any tile raises an anchored action row (Hide · Uninstall · Cancel); a gear button (top-left) opens a full-screen two-column list of all installed apps where each row's star toggles home visibility. Hidden state persists across restarts in a SQLDelight database; newly installed apps default to visible. The growth in wiring (database, repositories, ViewModel seams) is absorbed by introducing Hilt. The work ships as a five-PR Graphite stack — the user's first — so the plan doubles as a stacked-diff walkthrough.

Problem

Today the launcher shows every launchable app, unconditionally. There is no way to remove clutter (hide an app) or uninstall from the launcher, no settings surface at all, and no persistence layer to remember such choices. Architecturally, dependencies are hand-wired in LauncherViewModel.Factory; adding a database plus two new UI surfaces on that pattern would tangle construction logic further. Finally, the developer works on this repo through a headless remote session: the HTML spec itself needs a way to be seen (solved: published as a claude.ai Artifact — see Notes).

Solution

Five stacked phases, one Graphite branch/PR each. Phase 0 sets up the stack and lands this spec as the base PR. Phase 1 converts construction to Hilt (no behavior change). Phase 2 adds SQLDelight with a single hiddenApp table — favorite = absent from the table, so new installs are visible by definition — exposed as a Flow<Set<String>> and combined into LauncherState. Phase 3 builds the long-press action row: an overlay state machine in the ViewModel (Overlay.AppMenu), a scrim + anchored three-button row in Compose, Hide writing to the database with an animated grid reflow, and Uninstall firing the platform ACTION_DELETE intent (the system owns the confirmation dialog — we never build one). Phase 4 restructures the header to gear-left/clock-right and adds the full-screen two-column all-apps list with star toggles. Logic is verified with JVM unit tests; UI behavior is verified with journeys on a TV emulator (never Compose/Espresso tests, per CLAUDE.md convention).

[] Phase 2: SQLDelight visibility store (branch sqldelight-visibility, PR 3)

Persistence with the smallest possible schema: a hiddenApp table holding only hidden package names. Questionables

Hilt on AGP 9.2.1 — compatibility is the plan's biggest external risk

This project uses AGP 9.2.1 with AGP's built-in Kotlin (no kotlin-android plugin). The Hilt Gradle plugin historically lagged AGP majors; recent Dagger releases declare AGP 9 support, but this must be proven by the Phase 1 assembleDebug gate before anything is stacked on top — which is exactly why Hilt is the bottom code PR. If the newest Hilt still fails: fallback A is Dagger-without-the-gradle-plugin (feasible but loses @AndroidEntryPoint bytecode transform ergonomics), fallback B is deferring DI to manual constructor wiring in NocturneApplication (a ~40-line composition root that keeps every other phase intact). The stack structure means only PR 2 would be swapped.

Is SQLDelight + Hilt overkill for one set of package names?

Honestly: DataStore + manual wiring would cover today's feature in far less machinery. The user explicitly chose SQLDelight and Hilt as deliberate infrastructure investments for where the launcher is heading, so the plan honors that. The schema is kept to the minimum (one table, four queries) so the investment cost is mostly build wiring, which is exactly what the stacked PRs isolate. Assumption recorded; no action needed.

Schema shape: hidden-set table vs. per-app row with a flag

Chosen: store only hidden package names. "New installs default to shown" then requires no insert-on-detect logic, no sync between PackageManager and DB, and favorite-count = installed − hidden. A per-app table with isHidden would need reconciliation on every package change for zero present benefit. If future features need per-app metadata (sort order, launch counts), a new table can join against package name without migrating this one.

Reinstall semantics: pruning means uninstall→reinstall resets to visible

Pruning rows for uninstalled packages keeps the table honest and makes a reinstalled app behave like a new install (visible). The alternative — remembering hidden state across uninstall — seems user-hostile for a TV launcher (you reinstalled it, you presumably want to see it). Flagging because it's a one-line decision to flip: skip prune() and hidden state would survive reinstalls.

Uninstall flow: ACTION_DELETE + REQUEST_DELETE_PACKAGES, result via onResume

The design mandates the platform dialog. Intent.ACTION_DELETE with a package: URI triggers it; on API 26+ that requires the normal (auto-granted) REQUEST_DELETE_PACKAGES permission. Rather than registering ACTION_PACKAGE_REMOVED receivers, the plan leans on the existing ScreenResumed reload: the system dialog necessarily pauses the launcher, so returning always re-queries PackageManager and prunes. Limitation: apps uninstalled by other means while the launcher stays foregrounded (rare on TV) only disappear on next resume — accepted for now.

Long-press threshold: platform default instead of the design's configurable 300–1200ms

The design says 500ms, "tweakable 300–1200ms". Compose's onLongClick uses ViewConfiguration's long-press timeout, which is ~500ms — matching the spec's default with zero custom gesture code and correct auto-repeat handling. The tweakability reads as a prototyping affordance, not a product setting; building a custom hold-detector to make it configurable isn't worth the gesture-correctness risk. Revisit only if QA finds TV remotes triggering accidental long-presses.

Icons: unicode glyphs (⊘ ✕ ‹ ⚙ ★ ☆) instead of a material-icons dependency

The prototype itself uses unicode glyphs; the README says "replace with the product icon set", but this project has no icon set and minification is off, so material-icons-extended would be a heavy add for four glyphs. Newsreader may lack some of these codepoints — Android font fallback will render them from system fonts, which is visually acceptable at these sizes. If any glyph renders as tofu on the emulator, swap that one for a small hand-drawn ImageVector. Escalate to a proper icon dependency only if the user wants pixel-faithful visibility-off/delete icons.

Settings as an in-Activity overlay, not a navigation destination

The design frames settings as a "full-screen takeover" of the same surface, and the app currently has zero navigation library. Modeling it as Overlay.Settings in the existing single-state contract keeps the MVI-lite shape, keeps TV focus management in one tree, and avoids adding Navigation3 for one destination. If a third screen ever appears, that's the moment to introduce real navigation.

Stack base: reusing settings-rework as the spec PR

The branch already exists in this worktree, sits exactly at main, and its name matches the feature. Tracking it as the stack base (spec-only PR 1) avoids branch juggling in a worktree where main cannot be checked out. Cost: the PR is named after the branch, not docs/… — cosmetic. Alternative (fold the spec into the Hilt PR) rejected: a docs-only base PR is the gentlest first gt rep and gives reviewers the spec before the code.

Parallel branch scroll-fix carries CLAUDE.md and its own spec

The unmerged scroll-fix branch contains the repo's CLAUDE.md (source of the journeys-only testing rule) plus a scroll change to LauncherScreen.kt. This stack does not depend on it, but whichever lands second will need a small gt sync -f restack, and LauncherScreen.kt edits may conflict textually. Graphite primer for this stack (first-time reference)

Mental model: one branch = one PR = one reviewable idea, each branch parented on the previous. gt keeps children rebased when parents change.

MomentCommandWhat it does
Finish a phasegt create <branch> -am "msg"New branch on top of current, committing all changes (stages untracked too)
Push/refresh PRsgt submit --stack --no-interactive -dPushes every branch in the stack, opens draft PRs (-u = update-only later)
Fix review feedback on PR 2gt checkout hilt-di → edit → gt modify -a -m "address review"Amends that branch and auto-restacks PRs 3–5 on the new tip
Trunk moved / PR mergedgt sync -fPulls main, restacks everything, offers to delete merged branches
Rebase conflictfix files → git add . → gt continue(or gt abort, or gt undo to reverse the last gt op)
Look aroundgt log short · gt up/gt downSee the stack; walk between branches

Rules of the road in this repo: every gt command gets -m/--no-interactive/-q flags (they're interactive by default and will hang a non-interactive session); main is checked out in the sibling worktree, so never gt checkout main here; PR submission requires Graphite auth (gt auth --token …) — never bypass with git push.
